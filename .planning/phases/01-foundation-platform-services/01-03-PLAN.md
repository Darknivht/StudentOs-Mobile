# Plan 01-03: Supabase Auth with SecureStore Adapter, Session Persistence, Blocked User Detection

## Goal

Implement the complete authentication flow using Supabase Auth with `expo-secure-store` for encrypted token persistence, session persistence across app restarts, password reset via deep linking, sign out with secure cleanup, and immediate blocked user detection.

## Requirements

- **AUTH-01**: User can sign up with email and password
- **AUTH-02**: User can sign in and stay logged in across app restarts (expo-secure-store for tokens)
- **AUTH-03**: User can reset password via magic link email with deep linking
- **AUTH-04**: User can sign out from any page, clearing secure storage and redirecting
- **AUTH-05**: Blocked users are immediately signed out when is_blocked flag is detected

## Threat Model

| ID | Threat | Mitigation |
|----|--------|------------|
| T-1-01 | Auth tokens stored in plaintext SQLite | SecureStore adapter uses Android Keystore encryption |
| T-1-03 | Deep link token interception | Validate token format before processing, use HTTPS redirect URL in Supabase config |

## Tasks

### Wave 1

- [ ] **01-03-01**: Create SecureStore adapter for Supabase auth
  - Install `expo-secure-store`
  - Create `apps/native/services/secure-storage.ts`:
    ```typescript
    import * as SecureStore from 'expo-secure-store';

    export const ExpoSecureStoreAdapter = {
      getItem: (key: string) => SecureStore.getItemAsync(key),
      setItem: (key: string, value: string) => SecureStore.setItemAsync(key, value),
      removeItem: (key: string) => SecureStore.deleteItemAsync(key),
    };
    ```
  - **Handle size limit**: If a Supabase session exceeds 2KB (Android Keystore limit), split the session: store the refresh token in SecureStore (small), store the full session JSON in `expo-sqlite/kv-store` (encrypted at rest by Android full-disk encryption), and use the refresh token from SecureStore to validate.
  - **Verify**: Unit test confirms adapter methods call SecureStore APIs
  - **Req**: AUTH-02

- [ ] **01-03-02**: Create Supabase client factory in @studentos/shared
  - Create `packages/shared/src/supabase/client.ts`:
    ```typescript
    import { createClient } from '@supabase/supabase-js';
    import type { Database } from '../types/database';

    interface SupabaseConfig {
      url: string;
      anonKey: string;
      storageAdapter: SupportedStorage;
    }

    export function createSupabaseClient({ url, anonKey, storageAdapter }: SupabaseConfig) {
      return createClient<Database>(url, anonKey, {
        auth: {
          storage: storageAdapter,
          persistSession: true,
          autoRefreshToken: true,
          detectSessionInUrl: false,
        },
      });
    }
    ```
  - **Verify**: `createSupabaseClient` returns a typed Supabase client
  - **Req**: AUTH-02

- [ ] **01-03-03**: Initialize Supabase client in native app
  - Create `apps/native/services/supabase.ts`:
    ```typescript
    import { createSupabaseClient } from '@studentos/shared';
    import { ExpoSecureStoreAdapter } from './secure-storage';
    import { env } from '../lib/env';

    export const supabase = createSupabaseClient({
      url: env.EXPO_PUBLIC_SUPABASE_URL,
      anonKey: env.EXPO_PUBLIC_SUPABASE_ANON_KEY,
      storageAdapter: ExpoSecureStoreAdapter,
    });
    ```
  - **Verify**: `supabase` client initializes without error, SecureStore is configured
  - **Req**: AUTH-02

## Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Token storage | expo-secure-store (primary) + expo-sqlite/kv-store (oversized sessions) | Android Keystore for encryption; kv-store fallback for large JWTs |
| Session size handling | Split strategy: refresh token in SecureStore, full session in kv-store | Handles Android Keystore 2KB value limit |
| Blocked user check | Auth state change + 5-min poll + Realtime subscription | Triple redundancy ensures immediate detection |
| detectSessionInUrl | `false` | React Native doesn't use URL-based session detection; deep links handle password reset separately |

## Out of Scope

- OAuth/social login (not in web app)
- Biometric auth (v2)
- MFA/2FA (not in web app)
