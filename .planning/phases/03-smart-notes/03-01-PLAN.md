<threat_model>

## Trust Boundaries

| Boundary                    | Description                                                                    |
| --------------------------- | ------------------------------------------------------------------------------ |
| Client → Supabase notes API | Untrusted user input (note title, content, file uploads) crosses this boundary |
| Client → Supabase Storage   | File uploads (PDF, DOCX, images) cross this boundary                           |
| Note editor → Database      | Rich text HTML content stored — must be sanitized on read                      |

## STRIDE Threat Register

| Threat ID | Category            | Component              | Disposition | Mitigation Plan                                                                                                                                  |
| --------- | ------------------- | ---------------------- | ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| T-03-01   | I (Tampering)       | Note CRUD API calls    | mitigate    | Validate user_id matches auth session before insert/update — use Supabase RLS policies (already enforced by Supabase)                            |
| T-03-02   | X (XSS)             | Rich text HTML content | mitigate    | Sanitize HTML on display using a whitelist of safe tags (b, i, u, h1, h2, ul, ol, li, code, pre, p, br) — strip script, iframe, on\* attributes  |
| T-03-03   | S (Spoofing)        | File upload            | mitigate    | Validate file MIME type before upload (PDF: application/pdf, DOCX: application/vnd.openxmlformats, Image: image/\*) — reject unexpected types    |
| T-03-04   | D (DoS)             | Note creation quota    | mitigate    | Enforce daily note quota per tier in client (useNoteQuota) — server-side enforcement via RLS policy is recommended but not in this phase's scope |
| T-03-05   | I (Info Disclosure) | File URL storage       | accept      | File URLs point to Supabase Storage with RLS — only the owning user can access; low risk                                                         |

</threat_model>
